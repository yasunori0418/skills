# メトリクス早見表

test-monitor でメトリクスを選ぶときの参照。各メトリクスは「定義・取得元・完了基準との
紐づけ方」を簡潔に示す。対象特性に応じて必要なものだけ採用する（全部載せない）。

## 進捗系

### テスト消化率（test execution progress）

- 定義: 計画したテストケースのうち実行済みの割合（実行済み / 計画数）。
- 取得元: テストランナーの実行結果（実行件数）と test-plan のケース数。
- 完了基準との紐づけ: 「高リスク項目の消化率 100%」等の進捗系 exit criteria の裏付け。

### テスト実行時間（test duration）

- 定義: スイート全体・主要ジョブの実行時間。
- 取得元: CI ジョブの所要時間・ランナーのタイミング出力。
- 紐づけ: フィードバックの速さ（開発体験）。急増は基盤の劣化サイン。

## 品質系

### カバレッジ（coverage）

- 定義: ライン / ブランチ / 関数など、テストがテスト対象を実行した割合。
- 取得元: カバレッジツール出力（例: lcov・coverage.py・go test -cover・JaCoCo）。
- 紐づけ: 「カバレッジ ≥ N%」の exit criteria。どの粒度（ライン/ブランチ）かを明示する。

### 失敗率（failure rate）

- 定義: 実行したテストのうち失敗した割合（失敗 / 実行）。
- 取得元: JUnit XML 等のテスト結果レポート。
- 紐づけ: 「未解決の重大欠陥 0 件」等の品質系 exit criteria の間接指標。

### flaky（不安定）率

- 定義: 同一条件で合否が揺れるテストの割合。再実行での結果変動から検出する。
- 取得元: リトライ付き実行の結果比較・CI の履歴。
- 紐づけ: 基盤自体の信頼性。flaky が高いと他メトリクスの信頼性が落ちる。

## 取得形式の目安

- ランナーは **JUnit XML** 形式の結果出力を持つことが多く、失敗率・件数の集計に使いやすい。
- カバレッジは **lcov / cobertura** 等の標準形式で出せると集計スクリプトを共通化しやすい。
- どの形式が実際に出せるかは手順 1 の CI/ツール構成調査で確かめてから選ぶ。
